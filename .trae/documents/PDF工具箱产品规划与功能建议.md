# PDF 工具箱 - 产品规划与功能建议

> 面向 OPC 一人公司的纯前端 PDF 工具产品路线图\
> **基于 Qwen 3.6 编程能力的务实评估**

***

## 一、项目现状分析

### ✅ 已有优势

1. **技术架构扎实**

   * 纯前端架构，零后端依赖，部署成本极低

   * 使用 pdf-lib + pdf.js 双引擎，覆盖编辑和渲染场景

   * Web Worker 架构处理大文件，60秒超时保护

   * PWA 支持，可离线使用

   * 发布-订阅状态管理，架构清晰

2. **功能矩阵完整（9大功能）**

   * 查看、合并、压缩、转图片、拆分、图片转PDF、水印、发票拼版、隐私遮盖

   * 工作流 Pipeline 引擎（支持批量处理和模板系统）

3. **UI/UX 设计精良**

   * CSS 变量主题系统（浅色/深色）

   * 响应式设计，移动端适配

   * 丰富的动画效果（骨架屏、进度条、成功/错误图标）

   * 隐私计数器、操作后引导等细节

4. **安全隐私特性突出**

   * 严格 CSP 策略

   * 页面关闭自动清除数据

   * 隐私免责声明

### ⚠️ 当前不足

1. **功能层面**

   * 缺少 PDF 页面管理功能（旋转、删除页面、重新排序）

   * 缺少 PDF 加密/解密功能

   * 压缩功能较弱（仅使用 useObjectStreams，缺少图像重采样）

   * 缺少批量添加页码功能

2. **UI/UX 层面**

   * 所有功能使用 Modal 弹窗，缺少独立页面视图

   * 缺少功能搜索/快速定位

   * 缺少最近使用记录/收藏功能

   * 缺少多语言支持（目前仅中文）

   * 缺少快捷键支持

3. **技术层面**

   * 缺少单元测试

   * 缺少性能监控

   * 缺少错误追踪系统

   * 依赖版本较旧（pdf.js 3.11, pdf-lib 1.17）

## 三、功能开发建议（务实版）

### 🔥 P0 - 核心功能增强（1-2周，Qwen 可独立完成）

#### 1. PDF 页面管理 ⭐⭐

**功能描述：**

* 旋转页面（90°/180°/270°）

* 删除指定页面

* 拖拽重新排序页面

* 提取指定页面为新 PDF

**技术实现：**

```javascript
// 使用 pdf-lib 的 PDFDocument 页面操作 API
const pdfDoc = await PDFDocument.load(bytes);
const pages = pdfDoc.getPages();
pages[0].rotate(degrees(90));
pdfDoc.removePage(index);
const [copiedPage] = await targetPdf.copyPages(sourcePdf, [pageIndex]);
```

**UI 设计：**

* 网格视图显示所有页面缩略图

* 支持多选、拖拽排序

* 右键菜单提供旋转/删除/提取操作

#### 2. PDF 加密/解密 ⭐⭐

**功能描述：**

* 为 PDF 添加密码保护

* 移除现有密码（需提供原密码）

* 设置权限密码（禁止打印/复制/编辑）

**技术实现：**

```javascript
const pdfDoc = await PDFDocument.load(bytes, { password: 'old' });
const encryptedBytes = await pdfDoc.save({
  userPassword: 'user123',
  ownerPassword: 'owner456',
  permissions: {
    printing: 'highResolution',
    copying: false,
  }
});
```

#### 3. 批量水印/批量页码 ⭐⭐

**功能描述：**

* 为多个 PDF 批量添加水印（复用已有水印模块）

* 批量添加页码（支持自定义格式：第 X 页/共 Y 页）

* 批量设置元数据

**技术实现：**

* 基于现有 `batch-pipeline.js` 扩展

* 页码功能使用 pdf-lib 的 `drawText` API

## 四、明确放弃的功能及原因

### ❌ OCR 文字识别（可搜索 PDF）

**放弃原因：**

1. Tesseract.js 体积庞大（\~4MB 核心 + \~10MB 语言包）
2. 中文识别率远低于商业方案（百度/腾讯 OCR API）
3. **生成"可搜索 PDF"需要字符级 bbox 坐标，Tesseract.js 不直接提供**
4. 双层 PDF 生成需要复杂的坐标映射、字体嵌入、编码映射
5. Qwen 生成代码成功率 <40%，调试成本极高

**替代方案：**

* 如果用户有 OCR 需求，可引导使用云端 API（但违背"零上传"理念）

* 或仅提供简单文字提取（不生成可搜索 PDF）

### ❌ PDF 转 Word/Excel/PPT

**放弃原因：**

1. 纯前端转换库（如 pdf2docx）质量极差
2. 商业方案（Adobe、ABBYY）依赖多年积累的排版引擎
3. 表格识别、复杂排版还原是业界难题
4. 即使专业团队也难以做好，一人公司不应投入

### ❌ 数字签名（PKI 体系）

**放弃原因：**

1. pdf-lib **不支持**真正的数字签名
2. 需要 WebCrypto API + 证书管理
3. 涉及 PKI 基础设施，纯前端无法实现
4. 法律效力的数字签名需要 CA 证书

**可实现替代：** 视觉签名（图像叠加），已列入 P1 功能

### ❌ PDF 对比

**放弃原因：**

1. 差异算法复杂（文本对比、图像对比、布局对比）
2. 渲染一致性难保证（不同 PDF 阅读器渲染差异）
3. 用户需求低频，投入产出比低

## 十、风险提示

1. **浏览器兼容性**

   * Web Worker 支持（IE 不支持）

   * Canvas API 限制

   * 内存限制（移动端 \~500MB）

2. **法律风险**

   * 隐私遮盖功能的免责声明需完善

   * 视觉签名不具备法律效力，需明确提示用户

3. **技术风险**

   * pdf-lib 维护频率降低

   * pdf.js 大版本升级可能破坏兼容

   * 增强压缩功能可能遇到边界情况（特殊字体、加密 PDF）

## 十一、Qwen 3.6 使用策略

### 🟢 适合 Qwen 独立完成的任务

1. **API 组合调用**：文档完善的库（pdf-lib 基础功能）
2. **UI 组件开发**：HTML/CSS/JS 组件、响应式布局
3. **数据处理**：数组操作、状态管理、事件流
4. **标准功能**：搜索、快捷键、国际化、测试用例

### 🟡 需要人工审查的任务

1. **异步并发控制**：Worker 池、内存管理
2. **边界条件处理**：大文件、损坏文件、特殊编码
3. **性能优化**：代码分割、懒加载、缓存策略
4. **复杂逻辑**：图像重采样、表单填写

### 🔴 不建议交给 Qwen 的任务

1. **专业领域知识**：PDF 规范（ISO 32000）、字体子集化、色彩空间
2. **复杂调试**：内存泄漏、渲染异常、兼容性问题
3. **架构设计**：模块解耦、扩展性设计、技术选型

### 产品定位

**保持"纯前端、零上传、隐私优先"的核心价值主张**，在此基础上逐步增强功能矩阵，打造差异化竞争优势。

### 护城河的真正来源

❌ **不是这些功能本身**

* PDF 页面管理、加密等功能，任何开发者都能实现

* OCR、转 Word 等技术，商业方案已经非常成熟

✅ **真正的护城河是**

1. **用户体验**：极致的加载速度、流畅的交互设计、完善的错误提示
2. **隐私信任**："零上传"是差异化优势，开源透明可审计
3. **场景深耕**：发票拼版（已有）是好的垂直场景，继续挖掘中国用户特有需求
4. **工作流整合**：Pipeline 系统是亮点，强化"组合操作一键完成"的能力

### 一句话总结

**Qwen 3.6 是强大的编程助手，但不是魔法。用它来做"80 分的活"效率极高，但"95 分的活"仍需专业开发者。一人公司应聚焦核心优势，做深不做广，承认技术边界，不盲目追求"大而全"。**
